# Техническое задание на новую структуру Universal Converter

## 1. Статус документа

- Документ подготовлен по текущему состоянию проекта на 17.05.2026.
- Документ фиксирует новую фактическую структуру корневого SPA-приложения в каталоге проекта.
- Документ не заменяет старые ТЗ полностью, а уточняет и формализует именно новую структуру интерфейса, архитектуры и поведения после переработки вкладки валют.

## 2. Объект ТЗ

Объектом настоящего ТЗ является корневое веб-приложение Universal Converter, расположенное в файлах:

- `index.html`;
- `app.js`.

Справочные и унаследованные части workspace:

- каталог `course-convertor` используется как визуальный и UX-ориентир для валютного сценария;
- каталог `legacy_cpp` сохраняется как архив desktop-реализаций и референс предметной логики;
- основным пользовательским продуктом считается именно корневой SPA-модуль.

## 3. Назначение продукта

Необходимо поддерживать единый статический сайт-конвертер, в котором несколько категорий пересчета работают в одном интерфейсе без сборки и без серверного рендера.

Продукт должен решать следующие задачи:

1. Быстро пересчитывать физические величины и валюты.
2. Давать единый визуальный интерфейс для всех категорий.
3. Поддерживать расширяемую архитектуру категорий через реестр и адаптеры.
4. Сохранять пользовательские настройки локально.
5. Предоставлять отдельный встроенный калькулятор во вкладке валют без связи с самим валютным конвертером.

## 4. Новая структура проекта

Актуальная пользовательская веб-структура должна трактоваться так:

| Узел | Расположение | Роль |
| --- | --- | --- |
| Root SPA Shell | `/index.html` | HTML-каркас, CSS, контейнеры интерфейса, модальные окна |
| Root SPA Logic | `/app.js` | Вся логика категорий, рендер, состояние, локальное хранение |
| UX Reference | `/course-convertor` | Референс по стилю, настройкам, валютному UX |
| Legacy Reference | `/legacy_cpp` | Архивные Qt/C++ решения, не основная веб-ветка |
| Документация | `/README.md`, `/PROJECT_TZ.md`, `/UNIVERSAL_CONVERTER_TZ.md`, настоящий документ | Описание продукта и требований |

## 5. Архитектурная модель приложения

Приложение должно быть реализовано как SPA без шага сборки.

Архитектурные требования:

1. Основная логика должна быть сосредоточена в `app.js`.
2. Категории должны регистрироваться через единый `categoryRegistry`.
3. Каждая категория должна описываться объектом с параметрами:
   - `id`;
   - `name`;
   - `shortName`;
   - `adapter`;
   - `defaultValue`;
   - `defaultSourceUnit`;
   - `defaultActiveUnits`;
   - `units`.
4. Механика конвертации должна быть разделена по адаптерам, а не через несвязанные отдельные скрипты.
5. Интерфейс должен рендериться динамически в зависимости от активной категории.
6. Для валютной категории должна существовать собственная поверхность интерфейса, отличающаяся от общего шаблона физических величин.

## 6. Структура интерфейса

Корневой интерфейс должен состоять из следующих постоянных блоков:

1. Шапка приложения.
2. Переключатель темы.
3. Навигация по категориям.
4. Левая колонка ввода.
5. Правая колонка результатов или вспомогательного виджета текущей категории.
6. Toast-уведомление.
7. Модальное окно настройки валют.

### 6.1. Общая схема экрана

Для всех категорий должна использоваться двухколоночная схема:

- слева ввод и основные управляющие элементы категории;
- справа результаты пересчета или связанный с категорией дополнительный блок.

### 6.2. Схема для валютной категории

Во вкладке валют интерфейс должен быть разделен на два независимых блока:

1. Левый блок: валютный конвертер.
2. Правый блок: отдельный калькулятор.

Ключевое правило: правый калькулятор не должен управлять значениями конвертера и не должен использоваться как источник валютного пересчета.

## 7. Функциональная структура категорий

### 7.1. Категория «Давление»

Требования:

1. Категория должна использовать факторную модель конвертации через базовую единицу.
2. Базовой единицей должен быть Паскаль.
3. Пользователь должен вводить одно значение и выбирать исходную единицу.
4. Результаты должны отображаться карточками по остальным единицам.
5. Для каждого результата должна быть доступна кнопка копирования.

Поддерживаемые единицы:

- Па;
- кПа;
- МПа;
- бар;
- кгс/см²;
- атм;
- мм рт. ст.

### 7.2. Категория «Температура»

Требования:

1. Категория должна использовать отдельный температурный адаптер.
2. Должны поддерживаться шкалы:
   - Цельсий;
   - Фаренгейт;
   - Кельвин.
3. Пересчет должен выполняться автоматически после любого изменения значения или единицы.
4. Для температуры должны использоваться формулы со смещением, а не только коэффициенты.

### 7.3. Категория «Валюта»

Категория должна иметь отдельную структуру, отличную от обычного списка результатов.

Функциональные требования:

1. В левой колонке должен отображаться список активных валютных строк.
2. Пользователь должен иметь возможность вводить значение прямо в любую активную валютную строку.
3. После ввода в одной валюте остальные активные валюты должны пересчитываться автоматически.
4. Внутренней базовой расчетной валютой должен быть USD.
5. Формат отображения чисел должен использовать группировку тысяч по локали `ru-RU`.
6. В активном вводе должны сохраняться пробелы между тысячными разрядами во время редактирования.
7. Пользователь должен иметь возможность поднять валюту наверх отдельной кнопкой в строке.
8. Пользователь должен иметь возможность открыть модальное окно настройки валют.
9. В модальном окне пользователь должен иметь возможность:
   - включать и выключать валюты чекбоксами;
   - прокручивать длинный список;
   - менять порядок активных валют перетаскиванием.
10. Порядок активных валют и состав активного списка должны сохраняться в `localStorage`.

Поддерживаемые валюты текущей версии:

- RUB;
- UZS;
- KZT;
- USD;
- EUR;
- GBP;
- TRY;
- CNY;
- JPY;
- AED.

## 8. Требования к встроенному калькулятору

Калькулятор является отдельным UI-виджетом внутри валютной вкладки.

### 8.1. Общие требования

1. Калькулятор должен находиться в правой колонке валютной категории.
2. По умолчанию калькулятор должен быть скрыт.
3. В закрытом состоянии вместо него должна отображаться кнопка `Открыть калькулятор`.
4. После нажатия на кнопку калькулятор должен открываться в той же правой колонке.
5. В открытом состоянии должна быть доступна кнопка `Скрыть`.

### 8.2. Независимость от конвертера

1. Калькулятор не должен менять значения валютного конвертера.
2. Валютный конвертер не должен менять содержимое калькулятора.
3. Калькулятор должен иметь собственное состояние, отделенное от `currencyBaseUsdAmount` и валютных полей ввода.
4. Поле калькулятора не должно зависеть от активной валютной строки слева.

### 8.3. Поле ввода калькулятора

1. Верхнее поле калькулятора должно быть реальным редактируемым `input`, а не только текстовым отображением.
2. Пользователь должен иметь возможность вводить значение как клавиатурой, так и кнопками keypad.
3. Значение в поле должно форматироваться с разделением тысяч пробелами.
4. Поле должно поддерживать:
   - очистку;
   - удаление последнего символа;
   - смену знака;
   - дробный ввод через запятую.

### 8.4. Структура keypad

Кнопки калькулятора должны иметь фиксированную раскладку и не менять порядок в зависимости от разрешения экрана.

Фиксированная схема:

| Ряд | Кнопки |
| --- | --- |
| 1 | `AC`, `±`, `⌫` |
| 2 | `7`, `8`, `9` |
| 3 | `4`, `5`, `6` |
| 4 | `1`, `2`, `3` |
| 5 | `0`, `0`, `,` |

Требования к поведению keypad:

1. Порядок кнопок должен быть одинаковым на мобильных устройствах и на десктопе.
2. Кнопка `0` должна занимать увеличенную ширину в нижнем ряду.
3. Кнопки должны обновлять только поле калькулятора.
4. Нажатия не должны теряться при смене фокуса.

## 9. Состояние приложения

В `app.js` должно использоваться единое клиентское состояние.

Минимально обязательные группы состояния:

1. Активная категория.
2. Тема интерфейса.
3. Значения и исходные единицы для физических категорий.
4. Список активных валют.
5. Порядок валют.
6. Базовая сумма в USD для валютного конвертера.
7. Черновое значение активного валютного ввода.
8. Состояние модального окна настроек валют.
9. Состояние открытия/закрытия калькулятора.
10. Независимое текущее значение калькулятора.

## 10. Локальное хранение

Должны использоваться следующие ключи локального хранения:

| Ключ | Назначение |
| --- | --- |
| `universal-converter-theme` | Сохраненная тема интерфейса |
| `universal-converter-active-category` | Последняя открытая категория |
| `universal-converter-active-currencies` | Активные валюты и их порядок |

Требования:

1. При повторном открытии приложения должна восстанавливаться тема.
2. При повторном открытии приложения должна восстанавливаться последняя активная категория.
3. При повторном открытии валютной вкладки должны восстанавливаться состав и порядок валют.
4. Состояние калькулятора допускается хранить только в оперативной памяти текущей сессии, без обязательной записи в `localStorage`.

## 11. Нефункциональные требования

1. Приложение должно запускаться как статический сайт без npm-сборки.
2. Приложение должно корректно работать в браузере при запуске через локальный HTTP-сервер.
3. Интерфейс должен быть адаптивным.
4. Светлая и темная темы должны поддерживаться единым стилевым слоем.
5. Верстка не должна ломаться на мобильных экранах.
6. Валютная модалка должна поддерживать внутреннюю прокрутку.
7. Элементы управления должны оставаться читаемыми на компактной ширине экрана.

## 12. Ограничения текущей версии

1. Корневое SPA не использует backend собственной разработки.
2. Для валютной вкладки в текущей реализации допускается использование локального набора курсов как предварительной модели.
3. Каталог `course-convertor` не является главным runtime-модулем продукта.
4. Legacy Qt-приложения не входят в основной пользовательский сценарий веб-версии.

## 13. Критерии приемки

Новая структура считается реализованной корректно, если выполняются следующие условия:

1. Корневое приложение запускается из `index.html` и `app.js` без шага сборки.
2. Категории `Давление`, `Температура` и `Валюта` доступны из единой навигации.
3. Давление и температура используют общий архитектурный подход через реестр категорий и адаптеры.
4. Валютная вкладка имеет собственную левую поверхность со списком валютных строк.
5. Пользователь может менять порядок валют и набор активных валют через модальное окно.
6. Числа в валютных полях отображаются в читаемом виде с разделением тысяч.
7. Калькулятор по умолчанию скрыт.
8. После нажатия `Открыть калькулятор` появляется отдельный вводимый калькулятор.
9. Калькулятор не изменяет значения конвертера.
10. Конвертер не изменяет значение калькулятора.
11. Раскладка keypad остается статичной на разных разрешениях.

## 14. Рекомендуемое развитие после фиксации структуры

Следующим этапом допускается развитие без слома настоящей структуры:

1. Подключение живых курсов валют в корневое SPA.
2. Вынос конфигурации категорий в отдельные модули.
3. Добавление новых категорий конвертации.
4. Расширение калькулятора до арифметических операций с независимым вычислительным движком.
5. Добавление автотестов на ключевые сценарии интерфейса.
